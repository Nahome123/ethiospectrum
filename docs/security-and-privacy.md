# Security and privacy

Family, dependent, education, healthcare, benefit, and document information are sensitive. The current trust boundary is browser -> authorized server action/API -> Supabase with RLS -> private storage. Service-role credentials stay server-side. Signed URLs are short-lived; document content is never placed in source control, logs, analytics, test fixtures, or prompts beyond an authorized runtime request.

RLS is enabled on every private table. Household membership, explicit specialist assignment, and administrator checks govern access. Content editors can manage resources but cannot read household records. ETH-012 adds declared type, extension, filename, and size validation for private uploads. Security controls still planned before production include content-signature validation, malware scanning, production rate limits, CSRF review, headers, audit trails, retention/deletion workflows, backups/restoration testing, least privilege, and administrator MFA.

## Supabase credentials and privileged access

`NEXT_PUBLIC_SUPABASE_URL` and `NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY` are the only Supabase values allowed in browser configuration. The publishable key is not a bypass credential: database and storage access must remain constrained by RLS and storage policies. `SUPABASE_SECRET_KEY` is validated only in a `server-only` module and is never imported by browser utilities.

The elevated client in `lib/supabase/admin.ts` bypasses RLS. It is permitted only for reviewed administrative or background operations that cannot run under the requesting user’s RLS context. Do not use it in Client Components, ordinary route UI, or normal user-request flows; do not log it, serialize it, or expose it through `NEXT_PUBLIC_` configuration. Local development must use a local Supabase project and synthetic data, never a production project or production credentials.

Professional review is required for threat modeling, retention periods, incident response, regional/privacy obligations, HIPAA/FERPA/COPPA applicability, legal copy, and production penetration testing. This foundation makes no compliance claim.

## ETH-008 authentication controls

Supabase email/password authentication uses cookie-based PKCE sessions. Server authorization uses verified JWT claims through `auth.getClaims()` rather than `getSession()`, client state, or user metadata. The confirmation callback validates either a PKCE code or approved token-hash type without logging the code, hash, callback URL, cookies, access tokens, refresh tokens, or passwords. All return destinations must be supported-locale relative paths, preventing open redirects. A successful recovery callback creates only a short-lived, HTTP-only, locale-scoped recovery-intent cookie; both the reset page and update action require it, and a successful update clears it.

Password-recovery responses are neutral to avoid account enumeration. `NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY` is the only Supabase credential available to browser code. `SUPABASE_SECRET_KEY` remains in the `server-only` elevated client for reviewed background or administrative work and is not used for ordinary sign-in, sign-up, or recovery. CAPTCHA, production rate-limit monitoring, template review, CSRF/CSP review, and administrator role-governance require follow-up security review before production use.

## ETH-009 household and role controls

Profiles are self-readable and users may update only their own non-privileged fields. Application roles live in a separate table with no ordinary user write policy. Household and membership visibility requires active membership, and removal immediately removes access. Direct client membership creation, role escalation, ownership transfer, restoration, and hard deletion are denied. The public household RPC never accepts an owner ID, role, or permission from its caller.

Security-definer Auth synchronization and RLS helper functions use fixed empty search paths and fully qualified database objects to avoid search-path and policy-recursion attacks. They read metadata only for display fields and never for authorization. The `private` helper schema is not exposed through the Data API; only narrowly needed execution rights are granted to authenticated users. A service-role client remains prohibited from member-facing request flows.

## ETH-011 dependent controls

Dependent information is household-scoped and may be sensitive. `dependents` forces RLS and grants no anonymous access or delete permission. Active members and viewers can query only active dependents in their active household. Owners and household administrators alone may insert, update, or archive records. The archive operation writes `archived_at`; a trigger then prevents subsequent editing or household/creator reassignment. The normal member UI always filters archived profiles, including direct profile routes. Owners and administrators retain database visibility of archived records only because PostgreSQL evaluates the select policy for an update's proposed row; this is needed for the authorized archive transition and is not exposed through the active UI. Removal from `household_members` immediately removes every dependent read/write permission.

The server action accepts only permitted form fields and derives the household and actor from verified claims plus the active membership. It returns localized generic errors, never raw database errors. No dependent action imports `lib/supabase/admin.ts`, serializes a server secret, or trusts a browser-provided household or role.

## ETH-012 document controls

`family-documents` is a migration-defined private Storage bucket with a 20 MiB limit and allowlist for `application/pdf`, DOCX, and `text/plain` where Storage supports them. It has no public object URLs, and document code never calls `getPublicUrl`. The application also validates the declared MIME type, extension, non-empty size, maximum size, and normalized filename before preparing an upload. These checks improve safety and usability but do not prove the file contents are safe: this issue does not implement content-signature validation, malware scanning, OCR, parsing, or background processing.

The prepare action derives the user and active household from verified server-side claims and membership. It does not accept a household ID, uploader, document ID, or path from the browser. The database trigger creates the document UUID and final path, `households/{householdId}/dependents/{dependentId|unassigned}/documents/{documentId}/{safeFilename}`, validates an optional dependent belongs to the same active household, and locks identity and file metadata after preparation. An owner, household administrator, or member can prepare a pending document; a viewer, removed member, unrelated user, or anonymous user cannot. Active household members can read non-archived metadata for their household. Owners, administrators, and an eligible original non-viewer uploader retain only the archived-row visibility PostgreSQL needs to authorize their archive transition; member UI does not expose archived rows as active documents.

Storage object policies require a matching pending document row and exact generated path for an upload. They permit only the original active non-viewer uploader to create that object, prevent upsert replacement and arbitrary moves, and grant no normal-user object update or delete policy. Read access requires the exact uploaded, non-archived document and active household membership; a user removed from the household immediately loses it. The archive action is limited to an owner, administrator, or the original uploader who remains an active non-viewer member. It sets the database archive state only; a future retention workflow must decide when, whether, and under what approval to physically delete the private object.

The browser receives only the path, document ID, and time-limited signed-upload token needed for the direct upload; it never sends the file body through a Server Action. Tokens are not persisted or logged. Supabase upload tokens are valid for two hours and cannot be individually revoked, so a token issued before removal or archive could still place an orphan object at its exact path until expiry; it cannot complete or be read through the app after the database authorization check fails. Completion rechecks the expected object metadata before changing a pending row to `uploaded`, and upload failures are represented as `failed` rather than success. Pending rows left after an interrupted browser session and any such orphan objects are not automatically purged in ETH-012; scheduled cleanup is a future operational task. The authenticated download route validates the UUID and RLS-visible active uploaded row, creates a 60-second signed URL, and redirects to it. Missing, archived, unauthorized, and cross-household documents have the same safe not-found response. No member document flow imports or exposes `SUPABASE_SECRET_KEY` or the administrative client, and server responses keep database and Storage errors generic.
