# Document processing design

ETH-012 implements only the private upload boundary. Its current lifecycle is authenticated household user -> server-side metadata validation and pending document record -> Supabase signed upload token -> browser direct upload to `family-documents` -> server-side object metadata verification -> `uploaded` document record. The provider-issued upload token is a two-hour bearer capability; removal or archive after issuance prevents app completion and reads but cannot individually revoke a token that may still leave an orphan object at its one trusted path. `processing_status` remains `not_started`; there is no OCR, text extraction, DOCX parsing, language detection, embeddings, AI analysis, document chat, malware scanner, background job, or user notification in this issue.

The accepted-file allowlist is PDF, DOCX, and TXT at 20 MiB or less. Declared MIME type, extension, filename, and size checks are necessary input boundaries, but they do not prove a file's contents or safety. Content-signature validation and malware scanning are future hardening requirements and must run before any future processor treats an object as parseable. An interrupted upload may leave a pending or failed row; ETH-012 records the failure where practical but does not schedule abandoned-row or object cleanup.

ETH-013 adds a digital binder over existing document metadata only. It can search titles, normalized filenames, and categories; filter, sort, and paginate private records; and display the honest `upload_status` and `processing_status` values. It does not read object contents, extract text, add an index over content, claim that `not_started` processing occurred, enqueue a worker, or change the upload lifecycle. A binder search result is never evidence that a document has been scanned, parsed, OCR'd, translated, summarized, or reviewed.

Future processing begins only after an authorized, uploaded record: private object -> content-signature and malware checks -> idempotent background job -> page-level extraction -> language detection -> chunks and embeddings -> structured analysis and citations -> user notification. Workers must be idempotent, retryable, observable, permission-aware, duplicate-safe, resumable, and sanitized: no document contents, signed URLs, filenames, or raw private errors in logs or error telemetry. All derivatives must remain household- and document-authorized; a user denied the parent document is denied every derivative.
