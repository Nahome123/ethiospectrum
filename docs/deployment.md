# Deployment

Vercel-compatible deployment builds the Next.js application with environment variables supplied by the target environment. Keep development, staging, and production projects/secrets separate. Set only placeholders locally; server-only secrets must never be prefixed with `NEXT_PUBLIC_`. Before release, run the local database reset, pgTAP suite, database lint, generated-type refresh, and application checks. Review migrations in a non-production Supabase environment first; use `pnpm db:push:dry-run` to review a hosted change and run `pnpm db:push` only with explicit approval. Configure private storage, secure headers/session settings, rollback, and backup restoration before release.

ETH-014 additionally requires a distinct, high-entropy server-only `DOCUMENT_PROCESSING_SECRET` in each deployed environment. Configure a reviewed internal scheduler to `POST` the protected document-processing route with that value in `x-document-processing-secret`; do not use a URL parameter, request body, browser client, or `SUPABASE_SECRET_KEY` as a substitute. Deploy the migration and secret before enabling the scheduler. Monitor only aggregate job health and sanitized error-code counts. The route processes a bounded batch and must not be exposed as an unauthenticated public cron endpoint. Follow [document-processing-design.md](document-processing-design.md) for rotation, failure, and local-only verification procedures.

ETH-015 requires three additional server-only values in each reviewed environment: `OPENAI_API_KEY`, `OPENAI_SUMMARY_MODEL`, and a high-entropy `DOCUMENT_SUMMARY_SECRET`. The API key/model are used only by the server-side provider boundary. The summary secret protects only the internal summary worker and must not be reused from `SUPABASE_SECRET_KEY` or `DOCUMENT_PROCESSING_SECRET`. None may be prefixed with `NEXT_PUBLIC_`, committed, passed in a URL or browser request, or written to logs. Missing or partial provider configuration must fail closed.

Review the summary migration in a local or non-production Supabase environment, refresh generated types, and validate RLS before enabling any worker. A reviewed scheduler may invoke `/api/internal/document-summaries` with the summary secret in `x-document-summary-secret` and no document-specific body; it must receive aggregate counts only. Give that scheduler no repository-token permissions beyond its actual needs, never expose a public cron endpoint, and monitor only sanitized aggregate queue/state/failure data. This documentation does not assert that a hosted summary migration, provider configuration, or scheduler has been applied.

The repository includes `.github/workflows/document-processing.yml` for the approved GitHub Actions scheduler. It runs every 15 minutes and supports a manual run. Before merging it into the default branch, set these repository-level Actions values:

- Secret: `DOCUMENT_PROCESSING_SECRET` — the same distinct secret configured for the deployed app, never a Supabase key.
- Variable: `DOCUMENT_PROCESSING_ORIGIN` — the HTTPS deployed app origin, for example `https://ethiospectrum-web.vercel.app`, without a trailing path.

The workflow has no repository-token permissions and does not check out source code. It sends the secret only as an environment-backed request header. GitHub schedules use the workflow from the latest default-branch commit, so merge and deploy the application before relying on scheduled runs.

The repository's separate summary workflow follows the same least-privilege pattern. Before enabling it on a reviewed environment, configure a distinct `DOCUMENT_SUMMARY_SECRET` repository secret and the canonical HTTPS `DOCUMENT_SUMMARY_ORIGIN` variable. Do not substitute the processing secret, key, or workflow endpoint. Test it first against a non-production environment using synthetic data and a mocked provider; do not use scheduler logs as a place to inspect document text or generated summaries.

ETH-016 requires four additional server-only values in each reviewed environment: `OCR_PROVIDER` (currently `openai`), `OCR_API_KEY`, `OCR_MODEL`, and a high-entropy `DOCUMENT_OCR_SECRET`. The provider values must be configured together; partial or blank configuration fails closed. `DOCUMENT_OCR_SECRET` protects only `/api/internal/document-ocr`, must be sent only as `x-document-ocr-secret`, and must not reuse `SUPABASE_SECRET_KEY`, `DOCUMENT_PROCESSING_SECRET`, or `DOCUMENT_SUMMARY_SECRET`. Do not prefix any with `NEXT_PUBLIC_`, commit a real value, pass a document identifier/body/query string to the worker route, or log PDF bytes, page images, OCR text, provider payloads/responses, paths, or secrets.

Before enabling an OCR scheduler, deploy the migration and reviewed Node runtime dependency, refresh generated types, verify RLS and pgTAP against local synthetic data, and test the route with a mocked provider in a non-production environment. The scheduler should invoke only a bounded aggregate batch and needs no repository checkout/token permissions beyond its real transport. Configure alerts only for aggregate queue age/state/sanitized error counts. Successful OCR remains an aid that may contain errors, not an accuracy, certification, or professional-advice claim. This documentation does not assert that a hosted OCR migration, provider credential, or scheduler has been applied.
