# Data model

`auth.users` is the identity source. `public.profiles` stores only editable display fields, locale, timezone, and timestamps; it does not duplicate passwords, sessions, tokens, or email. The Auth trigger creates missing profiles and `public.user_roles` rows with the default `member` role, and the migration backfills existing Auth users without changing existing roles.

`public.user_roles` isolates privileged application roles from user-editable profiles. `public.households` uses a soft-delete timestamp, a primary owner, and a creator. `public.household_members` joins an Auth user to a household with `owner`, `administrator`, `member`, or `viewer` permission and an `active`, `invited`, or `removed` lifecycle. An owner membership must be active. Household creation inserts the household and owner membership together through the database function; invitations and ownership transfer remain deferred.

`public.dependents` belongs to exactly one household. ETH-011 adds a non-null `created_by` Auth-user reference and nullable `archived_at` timestamp, while retaining only first name, optional last/preferred name, optional birth year, optional school district/grade level, and limited notes. Database constraints trim text, reject blank required names and oversized fields, and keep the birth year between 1900 and the current year. A trigger sets `created_by` to the authenticated actor on insert, prevents later household/creator reassignment, and makes archived records immutable. The active-list index covers `household_id, created_at` where `archived_at is null`. A dependent with linked document metadata cannot be hard-deleted; normal workflows archive it instead so document assignment remains immutable.

ETH-012 makes `public.documents` the metadata authority for private uploads. A row has its UUID, household and optional dependent, authenticated uploader, trimmed title, normalized original filename, fixed `storage_bucket` (`family-documents`), generated `storage_path`, declared MIME type, file size, optional category and detected language, processing status, upload status, timestamps, and nullable `deleted_at`. Title and filename have bounded lengths; MIME type is restricted to PDF, DOCX, and TXT; file size must be greater than zero and no more than 20 MiB; an optional category is one of `education`, `health`, `legal`, or `other`; and `(storage_bucket, storage_path)` is unique. The active-list and dependent indexes support household document pages without including archived rows.

`upload_status` is `pending`, `uploaded`, `failed`, or `archived`. `processing_status` begins as `not_started`; it does not imply OCR, extraction, scanning, or AI processing. A database trigger derives `uploaded_by`, forces the private bucket, normalizes the filename, generates the UUID-backed storage path, sets audit timestamps, verifies that an optional dependent is active in the same household, and makes identity and file metadata immutable after preparation. A pending row can become uploaded or failed only by its original active non-viewer uploader; managers may archive records but cannot finalize another member's upload. Archive sets both the archived status and a server-owned `deleted_at`; normal member workflows never hard-delete a document or its object. Pre-ETH-012 rows without lifecycle metadata are marked `failed` fail-closed rather than treated as verified private uploads, so a hosted rollout must inventory and deliberately migrate or re-upload any legacy document records.

ETH-013 uses those existing `documents` fields for metadata-only binder search, filters, sorting, and pagination; it does not add a migration. The binder treats `created_at` as the upload-record timestamp used for date filtering and ordering, not as a guaranteed completed-upload timestamp. Existing active-household, active-uploaded, and active-dependent indexes remain in place; no full-text, trigram, category, status, or additional pagination index was added without a measured need. Default binder queries require `deleted_at is null`; an explicit `upload_status = archived` view is still constrained by document RLS. Active dependent options exclude `archived_at` rows, while already linked document metadata remains immutable and may use a neutral archived-dependent label when profile RLS withholds the name.

The trusted object-path shape is `households/{householdId}/dependents/{dependentId|unassigned}/documents/{documentId}/{safeFilename}`. The browser cannot choose any of its trusted IDs or supply a storage path. Document pages, chunks, embeddings, and analyses must derive access from their parent document, so a user denied the document is denied every derivative. `vector(1536)` is a planned embedding storage shape; dimension must be reviewed alongside the chosen model.
